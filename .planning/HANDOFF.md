### Архитектура в одном абзаце

`mtbank/` — общее framework-agnostic ядро. Его точка входа `run_analysis(audio) -> dict`
импортируют **и** OpenWebUI Pipeline (`pipelines/mtbank_pipeline.py`), **и** FastAPI
(`api/main.py`). Поток: аудио → `asr/audio.py` (скачивание/декод) → `asr/transcriber.py`
(faster-whisper) → `asr/diarizer.py` (Оператор/Клиент, только numpy) → `agents/supervisor.py`
(fan-out 4 агентов через ThreadPoolExecutor) → слияние в JSON-контракт из ТЗ.

## Задача: два бонуса (+10 баллов)

### Бонус A — Агент трендов (+5)

> ТЗ: «Агент трендов — анализирует несколько звонков, выявляет паттерны»

**Замысел:** пятый агент, который принимает список результатов `run_analysis()` по нескольким
звонкам и находит закономерности: доминирующие темы, динамику `quality_score`, повторяющиеся
нарушения compliance, типовые action items.

**Куда встроить:**
- `mtbank/agents/trends.py` — по образцу существующих агентов (`NAME`, `run()`, `fallback()`)
- Новый вход: REST `POST /analyze-batch` (список URL/файлов) → анализ каждого + сводка трендов
- В чате: если сообщение содержит **несколько** аудио-ссылок → батч-режим
- Unit-тесты по образцу `tests/test_agents.py` (LLM **всегда** фейк — см. `tests/conftest.py`)

**Стоимость:** 0 ₽ (те же вызовы Groq).

### Бонус B — Grafana-дашборд (+5)

> ТЗ: «метрики: количество звонков, quality_score, топ тематик»

**Замысел:** `api` отдаёт метрики в Prometheus-формате, Prometheus их собирает, Grafana рисует.

**Куда встроить:**
- `mtbank/metrics.py` — счётчики: `calls_total`, `quality_score` (histogram),
  `topic_total{topic=...}`, `compliance_failed_total`, `analysis_duration_seconds`
- `GET /metrics` в `api/main.py` (библиотека `prometheus-client`)
- Инкремент метрик внутри `run_analysis()` — так покрывается и чат, и REST
- `docker-compose.yml`: сервисы `prometheus` и `grafana`, конфиги в `deploy/grafana/`
- Дашборд как provisioned JSON, чтобы поднимался сам, без ручной настройки

**Стоимость:** 0 ₽ (оба open-source). **Но +~350 МиБ RAM** → на 2 ГБ VPS уже не влезет,
понадобится 4 ГБ. На Mac через туннель — без разницы.

## Как работать с проектом

```bash
docker compose up -d                     # openwebui:3000, pipelines:9099, api:8000, files:8899, proxy:8080
docker compose exec api pytest           # 44 теста, ~5 с
docker compose exec api pytest -m "not slow"   # без модели и сети, 0.3 с
docker compose logs -f api               # JSON-логи по стадиям
```

Внешние ссылки тоже работают (`https://raw.githubusercontent.com/ib0gdan/speech-analytics/main/test_data/...`).

## Грабли, на которые уже наступили (не повторять)

1. **`pipelines` держит модуль в памяти.** После правок в `mtbank/` контейнер надо
   пересоздавать (`docker compose up -d --force-recreate pipelines`), иначе чат отдаёт старый код.
2. **`requirements:` во frontmatter пайплайна — голый список пакетов.** Inline-комментарий
   уходит в pip целиком, установка падает, и сервер **уносит файл в `pipelines/failed/`**,
   удаляя его с примонтированного диска. Тяжёлые зависимости вшиты в `Dockerfile.pipelines`.
3. **nginx резолвит апстримы один раз.** Пересоздание `api` меняло IP → прокси молча отдавал
   502. Вылечено `resolver 127.0.0.11` + переменные (`deploy/local/nginx-tunnel.conf`).
4. **Прогрев обязателен.** Без него первый запрос грузит модель ~70 с и вылетает за лимит ТЗ.
   `warmup()` вызывается в `on_startup` пайплайна и в `lifespan` FastAPI.
5. **Не хардкодить время/цифры «на глаз»** — в этом проекте каждое утверждение подкреплено
   замером. Держать планку.
6. **Коммиты только от пользователя.** Никаких `Co-Authored-By: Claude`.
