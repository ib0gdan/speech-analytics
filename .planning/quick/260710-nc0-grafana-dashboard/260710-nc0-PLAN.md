## Design decisions (resolved before planning — implement these, do not re-litigate)

**Why the increment lives in `run_analysis`, not the API layer (design A + the whole point of the
shared core).** `pipelines` (chat) and `api` (REST) are SEPARATE processes with SEPARATE metric
registries. `run_analysis` is the one function BOTH import. Instrumenting there is the only single
point that covers chat + REST + batch. Instrumenting in the api layer would silently miss the chat
path — the primary graded interface. So: one call to `metrics.record_analysis(...)` at the end of
`run_analysis`, after the result dict is built.

**Two scrape targets, summed in PromQL (design A).** Because the two processes have independent
in-memory registries, the true total only exists in the dashboard: Prometheus scrapes BOTH
`api:8000/metrics` and `pipelines:9100/metrics`, and every panel uses `sum(...)` /
`sum by (topic)(...)` to merge them. The pipelines process has no route we can register on
OpenWebUI's own FastAPI app, so it starts a tiny in-process exporter via
`prometheus_client.start_http_server(9100)` in its existing `on_startup` hook — same process, same
registry that `run_analysis` increments. No pushgateway, no multiprocess mode (both processes run a
single uvicorn worker, so one process = one registry; state this assumption honestly in README).

**prometheus_client install locations + graceful-optional import (design B).** `mtbank.metrics` is
imported by the shared core, so `prometheus_client` becomes a hard import path for the api
container, the pipelines container, AND the test env. Install it in `api/requirements.txt` (covers
api + tests, which run inside the api container) and in `Dockerfile.pipelines` (baked in). Do NOT
add it to the pipeline frontmatter `requirements:` line — HANDOFF grabli #2: that line is fed to
pip verbatim and a bad line quarantines the pipeline into `pipelines/failed/`. Additionally,
`mtbank/metrics.py` guards the import in try/except: if the lib is ever absent, every `record_*`
and the exporter become no-ops so observability can never sink the core audio→JSON path (matches
the project's warmup/agent-fallback resilience style).

**Bounded label cardinality (design C).** `topic` is whitelisted to 5 values by the classifier
(кредиты, карты, переводы, жалобы, другое). `record_analysis` clamps any unexpected topic to
"другое" defensively. NO unbounded labels — never request_id, url, or filename as a label.

**Histogram for quality_score, NOT a summary/gauge (design D).** The score is a weighted sum of 4
fixed boolean checks (weights 20/30/30/20), so its support is the small discrete set
{0,20,30,40,50,60,70,80,100}. A Gauge would only show the last value. A Summary computes quantiles
per-instance and CANNOT be aggregated across the two scrape targets — a known Prometheus
limitation. A Histogram's buckets are counters that DO sum across targets (`sum by (le)`), and the
mean is `sum(..._sum)/sum(..._count)`. Buckets `(20,40,60,80,100)` cleanly separate the discrete
values. `analysis_duration_seconds` is also a Histogram, buckets `(5,10,15,20,30,45,60,90,120)`.

**RAM: opt-in via compose profile (design E).** prometheus+grafana go ONLY into `docker-compose.yml`
behind `profiles: ["metrics"]`. Plain `docker compose up -d` stays at today's ~2.3 GiB. The
reviewer opts in with ONE command: `docker compose --profile metrics up -d --build`.
`docker-compose.prod.yml` (the 2 GB VPS deploy) is NOT touched — Grafana never rides on prod.

**Batch falls out for free (design F) — verified, not assumed.** `mtbank/batch.py:69` calls
`run_analysis(...)` per source, so each successful batch call hits `record_analysis` exactly once;
the trends agent itself never calls `run_analysis`, so no double count. Failed sources raise
`AnalysisError` inside `run_analysis` before the record point, so they are not counted — consistent
with single-call semantics. A test asserts the +N delta.
